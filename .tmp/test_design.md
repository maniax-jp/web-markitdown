# テスト設計書: Web-MarkItDown

## 1. テスト方針
すべての変換処理をクライアントサイドで完結させるため、実際のファイルアップロードからダウンロードまでのエンドツーエンド (E2E) テストを中心に検証する。

## 2. テストケース

### 2.1 機能テスト (正常系)
| ケースID | テスト内容 | 期待結果 |
| :--- | :--- | :--- |
| TC-1 | PDFファイルのアップロードと変換 | 正確にテキストが抽出され、Markdown形式で出力されること。 |
| TC-2 | DOCXファイルのアップロードと変換 | 書式が維持され、Markdown形式で出力されること。 |
| TC-3 | XLSX/PPTXファイルのアップロードと変換 | 表形式やスライド内容が適切にMarkdown化されること。 |
| TC-4 | HTMLファイルのアップロードと変換 | HTMLタグが適切にMarkdown記法に変換されること。 |
| TC-5 | Markdownファイルの保存 | 変換後のテキストが `.md` ファイルとして正しくダウンロードされること。 |

### 2.2 機能テスト (異常系)
| ケースID | テスト内容 | 期待結果 |
| :--- | :--- | :--- |
| TC-6 | 非対応形式のファイルアップロード | 「対応していないファイル形式です」というエラーメッセージが表示されること。 |
| TC-7 | 破損したファイルのアップロード | Python側で例外が発生し、UIに適切なエラー通知が表示されること。 |
| TC-8 | 極端に大容量ファイルのアップロード | ブラウザがクラッシュせず、タイムアウトまたはメモリ制限のエラーが適切に処理されること。 |

### 2.3 UX/パフォーマンステスト
| ケースID | テスト内容 | 期待結果 |
| :--- | :--- | :--- |
| TC-9 | 初期ロード中の表示 | Pyodideロード中にスピナーが表示され、ユーザーに待機状態が伝わること。 |
| TC-10 | 変換中のUI応答性 | 変換処理中にブラウザのタブがフリーズせず、他の操作（キャンセル等）が可能であること。 |
